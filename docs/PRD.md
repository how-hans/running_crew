# PRD (Product Requirements Document) — AgriSage

Agriculture + Sage. "We don't just name the problem. We walk you through it."

- 작성일: 2026-07
- 팀: Team C — Hyunjun Lee, Hanmin Bae, Minseong Hong, Najin Son, Sehyeon Kim
- 문서 상태: Draft v1 (1차 발표 이후 업데이트)

## 1. 개요 (Overview)

AgriSage는 작물 사진 한 장으로 병충해를 진단하고, 그 판단 근거를 설명하며, 사용자 상황에 맞는 방제 방법을 추천하고, 방제 이후까지 확인해주는 AI 기반 농업 지원 서비스다. 기존 서비스들이 "진단 정보 제공"에서 멈추는 지점을 넘어, 진단부터 사후관리까지 하나의 흐름으로 완결하는 것을 목표로 한다.

## 2. 문제 정의 (Problem Statement)

### 2.1 배경

- 2025년 귀농 인구는 전년 대비 8.5% 증가했으나, 도시로 재이주한 귀농인 중 73.8%가 정착 후 3년 이내에 이주를 결정함 — 초기 정착 구간이 가장 취약함
- 온라인 식물 상담 데이터에서 상담 문의의 79%가 실제 병충해가 아닌 재배 관리·생리 장해 문제였음 — 초보자는 "이게 병인지 아닌지" 자체를 판단하기 어려움
- 농약 오사용은 초보자뿐 아니라 숙련 농민에게도 흔함 (미등록 작물 사용, 잘못된 살포 간격, 과도한 농도 등) — 국립농산물품질관리원 조사에서 약 4.4만 건 중 777건이 기준 초과 판정
- 국제 표준 안전기준(Positive List System, PLS)을 준수하지 않으면 잔류허용기준이 일률기준(0.01ppm)으로 적용되어 수확물 전체가 폐기 대상이 될 수 있음

### 2.2 핵심 문제 (Two-Stage Problem)

| 단계 | 문제 |
| --- | --- |
| Stage 1: 진단 | 병인지, 과습인지, 영양 부족인지 스스로 판단하지 못함 |
| Stage 2: 치료 | 진단이 맞아도 어떤 제품을, 어떻게, 언제 써야 하는지 판단 기준이 없음 |

## 3. 목표 (Goals)

### 3.1 제품 목표

- 사진 한 장으로 신뢰할 수 있는 1차 진단 제공
- 진단 근거를 사용자가 이해할 수 있는 언어로 설명
- 사용자 상황(재배 목적, 유기농 여부 등)에 맞춘 방제 제품 추천
- 안전기준(PLS) 자동 체크로 사용자 실수 예방
- 방제 후 결과를 확인하는 폐루프(closed-loop) 제공

### 3.2 성공 지표 (제안, 추후 구체화 필요)

- 모델 분류 정확도 (Accuracy, F1 Score)
- 진단 결과에 대한 사용자 이해도/만족도 (설문 기반)
- 방제 후 재방문율 (사후관리 기능 사용률)
- 안전기준 경고가 실제로 노출되고 인지된 비율

## 4. 타겟 사용자 (Target Users)

| 구분 | 설명 |
| --- | --- |
| Primary | 귀농·귀촌 후 첫 3년 이내인 초보 재배자 — 기본적인 재배 판단력을 아직 쌓지 못한 상태이며, 모든 결정이 부담스러운 시기 |
| Secondary | 소규모·취미 재배자 — 텃밭, 주말농장, 베란다 재배 등 규모는 작지만 동일한 불확실성을 겪는 사용자 |

공통 니즈: "지금 보고 있는 것이 무엇인지, 다음에 무엇을 해야 하는지"에 대한 확신

## 5. 범위 (Scope)

### 5.1 In Scope (이번 버전에 포함)

- 사진 기반 작물 질병 분류 (38개 작물-질병 조합)
- LLM 기반 진단 결과 설명 생성
- 사전 구축된 질병-농약 매핑 데이터베이스 기반 방제 제품 추천
- 수확 예정일 vs 농약 안전사용기간(PLS 기준) 자동 비교 및 경고
- 방제 후 사후 확인 알림 (재사진 요청 → 개선 여부 판단)

### 5.2 Out of Scope (이번 버전에서 제외)

- 원클릭 결제/구매 연동 (법적 리스크 및 개발 범위 고려하여 제외, 추천까지만 제공)
- 대화형 문진(멀티턴 챗봇으로 confidence 낮은 케이스를 되물어 좁히는 기능) — 로드맵 항목으로 이관
- 사투리 대응 음성인식, 카카오톡 챗봇 배포, ARS 전화 상담 등 접근성 확장 기능
- 마을/농협 단위 대시보드, 가족 연동 알림

## 6. 기능 요구사항 (Functional Requirements)

### FR-1. 이미지 진단

- 사용자가 작물 잎 사진을 업로드하면, 학습된 이미지 분류 모델이 38개 클래스 중 하나로 분류
- 분류 결과와 함께 confidence score 제공

### FR-2. 결과 설명 (Explainable Diagnosis)

- 분류 결과(작물명, 질병명)를 LLM에 전달
- LLM은 새로운 진단을 내리지 않고, 확정된 분류 결과를 기반으로만 설명 생성
- 출력 항목: 진단 결과 / 병의 특징(사용자 눈높이에 맞춤) / 발생 원인 / 추천 대응법

### FR-3. 맞춤형 방제 추천

- 사전 구축된 질병-농약 매핑 DB에서 해당 질병에 유효한 제품 후보 추출
- 사용자 조건(재배 목적, 유기농 여부 등, 추후 구체화)에 따라 후보를 압축

### FR-4. 안전기준(PLS) 체크

- 사용자가 입력한 예상 수확일과, 추천 제품의 안전사용기간(pre-harvest interval)을 비교
- 충돌 시 경고 메시지 및 대체 제품 안내

### FR-5. 사후관리 (Follow-up)

- 방제 안내 후 일정 기간(예: 3~5일) 뒤 사용자에게 재확인 알림 발송
- 재사진 업로드 시 개선 여부 판단, 개선 안 될 경우 원인 재진단 유도

## 7. 사용자 흐름 (User Flow)

```
사진 촬영/업로드
      ↓
이미지 분류 모델 (InceptionV3, Transfer Learning) → 작물-질병 분류 + confidence
      ↓
LLM 기반 결과 설명 생성
      ↓
질병-농약 DB 매칭 → 맞춤형 방제 제품 추천
      ↓
PLS 안전기준 자동 체크 (수확 예정일 vs 안전사용기간)
      ↓
[방제 실행]
      ↓
N일 후 사후 확인 알림 → 재사진 요청 → 개선 여부 판단
```

## 8. 기술 요구사항 (Technical Requirements)

### 8.1 모델

- 아키텍처: InceptionV3 (GoogLeNet 계열), ImageNet Pretrained
- 학습 방식: Transfer Learning + Fine-tuning
  - 초반 저수준 특징 레이어는 freeze
  - 후반 레이어 + 신규 38-class 출력층은 재학습
- 선정 근거: PlantVillage 데이터셋 원 논문(Mohanty, Hughes & Salathé, 2016)에서 60개 실험 구성 비교 결과, GoogLeNet + 전이학습 + 컬러 이미지 조합이 최고 정확도(99.34%) 기록
- 평가 지표: Accuracy, F1 Score, Confusion Matrix

### 8.2 데이터

| 항목 | 내용 |
| --- | --- |
| 원본 데이터 | PlantVillage (Hughes & Salathé, 2015), 54,306장, 통제된 환경 촬영, 배경 단일 |
| 사용 데이터 | New Plant Diseases Dataset (Kaggle, 증강), 87,000장+ |
| 클래스 수 | 14개 작물 × 26개 질병 유형 조합 → 38개 고유 클래스 (동일 질병이라도 작물이 다르면 별도 클래스) |
| 데이터 분할 | Train / Validation / Test (구체 비율 TBD, 원 논문 기준 80-20이 최고 성능) |

### 8.3 LLM 파이프라인

- 입력: 확정된 분류 결과(작물명, 질병명)만 전달 — LLM이 이미지 자체를 재진단하지 않음
- 출력: 구조화된 설명(진단 결과 / 원인 / 대응법 / 안전 정보)
- 농약 추천 및 안전사용기간 정보는 LLM이 생성하지 않고, 사전 구축된 확정 DB 값을 기반으로 함 (환각 방지)

### 8.4 알려진 한계 (원 논문 기반)

- 통제된 환경(단일 배경, 정면 촬영)에서 학습된 모델은 실사용 환경(다양한 배경, 각도)에서 정확도가 크게 하락할 수 있음 (원 논문에서 31%까지 하락 사례 보고)
- 이를 보완하기 위해 confidence가 낮은 경우를 명확히 표시하고, 사후관리 루프로 오진단을 사후에 보정하는 설계 필요

## 10. 리스크 및 가정 (Risks & Assumptions)

| 항목 | 내용 |
| --- | --- |
| 모델 일반화 리스크 | 통제된 환경 데이터로 학습된 모델이 실제 사용자 사진(다양한 조명/각도/배경)에서 정확도가 낮을 수 있음 |
| LLM 환각 리스크 | 농약 추천/안전 정보는 반드시 확정 DB 기반으로만 생성하도록 강제해야 함 (LLM이 임의로 만들어내면 안전사고 위험) |
| 데이터베이스 구축 리스크 | 질병-농약 매핑 DB의 실제 구축 범위와 정확도가 아직 미정 — 우선순위 높은 작업 |
| 법적 리스크 | 농약 직접 판매는 법적 규제 대상이므로, 이번 버전은 추천까지만 제공하고 구매 연결은 하지 않음 |

## 11. 로드맵 (향후 고려사항)

- 대화형 문진(confidence 낮은 케이스를 LLM이 되물어 좁히는 기능) 도입
- EfficientNet 등 최신 아키텍처와의 성능 벤치마킹
- 실사용 환경(비통제 환경) 데이터로 모델 강건성 보완
- 접근성 확장: 음성 안내, 카카오톡 챗봇 배포 등 (초기 아이디어 단계, 이번 버전 범위 외)
